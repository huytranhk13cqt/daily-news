### 🔭 Overview (Tổng)
Apache Airflow 3.2.0 was released on 07/04/2026 as the first minor release after the 3.x line stabilised last year, and it is the release that operationalises the data-asset model introduced in 3.0 rather than just polishing it. The headline features are Asset Partitioning, Multi-Team Deployments, and synchronous callbacks for Deadline Alerts, along with a new `--only-idle` scheduler flag and the migration of serialization logic into the Task SDK. For platform teams running Airflow as a shared service, this release is the one that lets them stop running one cluster per business unit and stop triggering whole DAGs on every asset refresh.

### 🔬 Details (Phân)
Asset Partitioning extends data-aware scheduling so that a downstream task is triggered by a specific partition of an asset rather than by any change to the asset at all. In practical terms, a daily sales DAG that writes `sales/2026-04-18` no longer needs to fan out into every downstream region-specific job on every run; only the consumers whose partition selector matches will fire. Multi-Team Deployments introduce isolated tenants within one Airflow installation, with each team getting its own DAGs, connections, variables, pools, and executors, and with RBAC scoped accordingly. Deadline Alerts now support a `SyncCallback` alternative to the original async callback, executed by the executor rather than the triggerer and optionally pinned to a named executor via the `executor` parameter; a DAG can also pass a list of deadline specs to register multiple alerts per run.

Two of these features — Multi-Team Deployments and Deadline Alerts — are still marked experimental in 3.2.0, which matters for adoption planning. The release arrives alongside a 3.2.1 release candidate posted on 15/04/2026, indicating the release cadence is back on a fortnightly bugfix rhythm. Teams on 3.0.x or 3.1.x can upgrade in place, but those still on 2.x should treat the 3.x migration as the prerequisite project rather than trying to pick up 3.2 features directly.

### 🎯 Takeaway (Hợp)
If a platform team has been waiting for partition-aware scheduling before moving off the 2.x `@daily` macro pattern, 3.2.0 is the version that closes that gap. The concrete next step is to pilot Asset Partitioning on one high-fanout DAG and read the Multi-Team Deployment migration notes before committing to it as a tenancy model, given its experimental flag.

### 💡 Example
```python
# Airflow 3.2.0 — asset partitioning: only sales for the matching region trigger downstream work.
from airflow.sdk import asset, dag, task
from airflow.sdk.definitions.asset import AssetRef

@asset(partitioning_keys=["region", "ds"])
def sales(region: str, ds: str) -> dict:
    return {"region": region, "ds": ds, "rows": 42}

@dag(schedule=[AssetRef("sales", partition={"region": "APAC"})])
def apac_rollup():
    @task
    def rollup(sales_partition):
        # Triggered only when a new APAC partition of `sales` appears.
        return sales_partition["rows"]

    rollup(AssetRef("sales"))
```

## 5. Raspberry Pi OS 6.2 Disables Passwordless sudo by Default on New Installs

### 🔭 Overview (Tổng)
Raspberry Pi OS 6.2 landed on 14/04/2026 as the second Trixie-based update, and it closes a default-configuration gap that security auditors have flagged for years: newly installed systems no longer ship with passwordless sudo enabled. Any new install from this release will prompt for the current user's password on each `sudo` invocation, with a five-minute authentication cache. Existing installations are untouched, and the new behaviour can be reverted through Control Centre, `raspi-config`, or a manual sudoers drop-in file.

### 🔬 Details (Phân)
The practical effect is that a Pi booted from a 6.2 image and plugged into a shared Ethernet port no longer hands local-root privileges to anyone who can reach its login prompt. The password cache length is compatible with the default behaviour on Debian and Ubuntu, so scripts that run a burst of `sudo` calls inside five minutes remain unaffected. The Admin Password switch is exposed in the Control Centre's System tab, which is the discoverable way for hobbyist users to flip the behaviour one way or the other; `raspi-config` offers the same toggle for headless setups, and advanced users can drop a `NOPASSWD: ALL` line into a sudoers file if they prefer configuration-as-code. Critically, existing Raspberry Pi OS 6.x installations retain passwordless sudo on upgrade — the switch is visible in Control Centre but stays in its legacy position.

This is a deliberate rollout strategy that prioritises backward compatibility for the large fleet of hobbyist devices that rely on unattended sudo in automation scripts, while applying the safer default to new devices entering the field. For IoT deployments that image Pis at scale, the correct action is to bake the desired posture into the golden image rather than relying on the new default: either keep the Control Centre default (password-on) for production devices, or explicitly re-enable passwordless sudo with a sudoers drop-in file for unattended kiosks and signage boxes where the prompt would break booting flows.

### 🎯 Takeaway (Hợp)
Fleet operators imaging Raspberry Pi devices in 2026 should update their provisioning scripts before the 6.2 image becomes the default NOOBS download, because silent assumptions about passwordless sudo will start failing on fresh installs. Solo hobbyists upgrading in place see no change — which is both the release's strength and its limitation.

### 💡 Example
```bash
# Inspect current sudo posture on Raspberry Pi OS 6.2.
sudo -n true 2>/dev/null && echo "passwordless sudo is ENABLED" || echo "password required"

# Re-enable passwordless sudo for a headless kiosk image via drop-in (use sparingly):
sudo install -m 0440 /dev/stdin /etc/sudoers.d/010_kiosk-nopasswd <<'EOF'
kiosk ALL=(ALL) NOPASSWD: ALL
EOF

# Or flip the Admin Password toggle from the CLI on 6.2:
sudo raspi-config nonint do_sudo 0   # 0 = passwordless, 1 = require password
```
