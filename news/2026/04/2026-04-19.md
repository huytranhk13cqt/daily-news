## 1. Apache Kafka CVE-2026-35554 Silently Misroutes Producer Messages

| Field | Value |
|---|---|
| 📅 Date | 17/04/2026 |
| 🏷️ Topic | Security |

| 🔑 Keywords | `apache-kafka` · `cve-2026-35554` · `buffer-pool-race` |
| ⚡ Impact | High |

### 🔭 Overview (Tổng)
Apache Kafka disclosed CVE-2026-35554 on 17 April 2026, a high-severity race condition in the Java producer's ByteBuffer pool that can silently corrupt messages and misroute them to unintended topics. The vulnerability carries a CVSS v3.1 score of 8.7 and affects `kafka-clients` from 2.8.0 up through a wide range of 3.x and 4.x builds, which between them cover most deployments currently in production. It is one of the more unusual Kafka bugs because no error is raised to the producer when the fault occurs, so pipelines can continue to report success while data crosses topic boundaries.

### 🔬 Details (Phân)
The race window opens when a batch expires through `delivery.timeout.ms` while its request is still in flight on the network. The expired batch's ByteBuffer is returned to the pool before the network layer has finished with it, and a subsequent batch destined for a different topic can pull the same buffer, leaving a corrupted payload addressed to the wrong partition. In environments where Kafka topics carry different security classifications — for example, PII traffic on one topic and public event data on another — the misrouted records are delivered to every consumer authorised on the destination topic, which is precisely the class of leak that ACLs and topic-level SerDes are supposed to prevent.

Fixed versions are 3.9.2, 4.0.2, 4.1.2 and 4.2.0 and later. Affected ranges are `kafka-clients >= 2.8.0 and < 3.9.2`, `>= 4.0.0 and < 4.0.2`, and `>= 4.1.0 and < 4.1.2`. Client-broker compatibility means producers can be upgraded independently of brokers, so there is no need to schedule a cluster-wide maintenance window to apply the fix. For branches older than 3.9 the only supported path is to move onto 3.9.2 or the 4.x series; raising `delivery.timeout.ms` reduces probability but does not eliminate the race and should not be treated as a workaround.

### 🎯 Takeaway (Hợp)
Patch producer libraries this week, not next sprint. The combination of "no error surfaced" and "data delivered to the wrong topic" means existing observability will not flag the issue, and affected records will not be in a dead-letter queue — they will already be on a topic someone else reads. Tác động sẽ nặng nhất ở các môi trường multi-tenant dùng Kafka ACLs làm ranh giới bảo mật.

## 2. CISA Adds 13-Year-Old Apache ActiveMQ RCE to the KEV Catalog

### 🔭 Overview (Tổng)
On 17 April 2026 the US Cybersecurity and Infrastructure Security Agency added CVE-2026-34197 to its Known Exploited Vulnerabilities catalog, triggering Binding Operational Directive 22-01 and giving federal agencies until 30 April 2026 to patch. The bug is a remote code execution flaw in Apache ActiveMQ Classic's Jolokia management API that, according to researchers at Horizon3, has been sitting in the codebase for about 13 years. ShadowServer is tracking more than 8,000 publicly reachable ActiveMQ instances, and Fortinet FortiGuard Labs reports exploitation activity peaking in mid-April.

### 🔬 Details (Phân)
The vulnerability technically requires authentication to the Jolokia endpoint, but the real-world exploitation path is simpler than that sounds. Many ActiveMQ deployments run with default `admin:admin` credentials on the web console, and on versions 6.0.0 through 6.1.1 the older CVE-2024-32114 still exposes the Jolokia API without authentication at all. Chained together, those two flaws produce an unauthenticated remote-code-execution pathway, which is why CISA escalated the new CVE rather than leaving it as a routine advisory. Patches are available in ActiveMQ 5.19.5 and 6.2.3.

Operationally this matters beyond US federal networks because ActiveMQ is widely used as the message backbone for JMS-based ETL and legacy event-driven systems, and those deployments frequently sit on the internal network without the same patching cadence as front-door infrastructure. The CISA listing is also useful as leverage for teams that have been trying to retire ActiveMQ instances: a KEV entry is a much clearer signal to management than an internal memo, and binding it to a 30-day federal deadline turns the conversation into a schedule rather than a priority debate.

### 🎯 Takeaway (Hợp)
Inventory every ActiveMQ broker, confirm the Jolokia endpoint is not exposed to anything you do not control, and move to 5.19.5 or 6.2.3 this sprint. If the broker is running 6.0.0 through 6.1.1, assume the unauthenticated chain applies and prioritise accordingly. Nếu data team đang trì hoãn việc chuyển off ActiveMQ, đây là cái cớ chính đáng để đẩy lịch di trú lên.

## 3. Raspberry Pi OS 6.2 Turns Off Passwordless sudo by Default

| Field | Value |
|---|---|
| 📅 Date | 17/04/2026 |
| 🏷️ Topic | Security |
| 🌐 Domain | IoT |
| 🔑 Keywords | `raspberry-pi-os` · `passwordless-sudo` · `control-centre` |
| ⚡ Impact | Medium |

### 🔭 Overview (Tổng)
Raspberry Pi OS 2026-04-13 — versioned 6.2 — ships with passwordless `sudo` disabled by default, a change that the Raspberry Pi Foundation explained on its news blog and that saw continued coverage through 17 April across 9to5Linux, Neowin, The Register and Hackster. For a platform whose default user experience has shipped with uncontested root-adjacent access since the project's earliest Debian images, this is a meaningful break from convention. It affects anyone fresh-flashing an SD card from this release onward; existing installations remain unchanged unless the user flips the new Control Centre toggle.

### 🔬 Details (Phân)
In the new default, running `sudo` at the terminal or through any Control Centre action that needs elevated rights now prompts for the account password, with a five-minute re-use window before the prompt appears again. Users who want the old behaviour can flip the Admin Password switch in the Control Centre System tab or use `raspi-config`, and the timeout is configurable by editing `/etc/sudoers` with `visudo` and setting `Defaults timestamp_timeout=X`. Upgrades preserve the previous state, so nobody's fleet of deployed nodes suddenly starts failing automated provisioning scripts.

Why this matters for IoT teams is scripting hygiene. A lot of Pi-based field equipment — signage, sensor gateways, edge gateways, kiosks — is provisioned with Ansible, cloud-init, or handwritten `curl | bash` installers that assume `sudo` never prompts. On new images those scripts now need either an `NOPASSWD:` line in a drop-in under `/etc/sudoers.d/`, an SSH key paired with an explicit service account, or a first-boot step that sets up the password-less behaviour before any other automation runs. The change also removes a long-standing piece of low-hanging fruit for local attackers with physical access to an unattended Pi.

### 🎯 Takeaway (Hợp)
Audit every provisioning script and golden image that assumes passwordless `sudo`, then decide whether to pin to the old behaviour in a small drop-in file or embrace the new default and refactor automation to use service-account SSH keys. Với các fleet đang triển khai ngoài hiện trường, nên kiểm tra tối thiểu một node mới flash 6.2 trước khi cho image mới vào sản xuất.

### 💡 Example
```bash
# Make Raspberry Pi OS 6.2 behave like the older releases for a single
# automation user, without reverting the system-wide default.
sudo tee /etc/sudoers.d/010_provision-nopasswd <<'EOF'
provision ALL=(ALL) NOPASSWD:ALL
EOF
sudo chmod 0440 /etc/sudoers.d/010_provision-nopasswd

# Or keep the password requirement and lengthen the grace window (in minutes).
sudo visudo
# Add or edit:  Defaults timestamp_timeout=15
```

### 🔭 Overview (Tổng)
Uber's engineering blog published "Open Source and In-House: How Uber Optimizes LLM Training" on 14 April 2026, detailing the distributed-training stack it runs on its Michelangelo ML platform. The write-up is unusually specific about which open-source components the team kept, which it extended, and what it built in-house to make large-model training and inference affordable at the company's scale. For data platform teams trying to decide whether to buy, rent, or build, this is one of the more useful reference architectures published this month because it names throughput and memory numbers alongside the tool choices.

### 🔬 Details (Phân)
The stack is PyTorch with DeepSpeed ZeRO (stages 1 through 3) for memory optimisation, Ray Train for distributed orchestration, Hugging Face Transformers for model APIs, NCCL for inter-GPU communication, and vLLM for inference. The post reports that DeepSpeed ZeRO stage 3 with CPU optimiser offload delivered at least a 34% reduction in GPU memory footprint and 2–3x throughput gains on their workloads, flash attention let them double batch sizes at broadly similar step times, and H100 clusters produced roughly 3x the inference throughput of A100 on Mixtral 8x7b. Network-attached storage pulled 10 GB/s on H100 and 3 GB/s on A100 for Llama 2 70B pre-training, which is the sort of bottleneck a lot of teams only notice after they have already bought the GPUs.

In-house tooling consists of checkpoint management with remote storage support (HDFS and Uber's Terrablob), an offline LLM scorer for batch prediction built on Ray and vLLM, and integrations into the company's feature store. Notably absent from the stack are Megatron-LM, NeMo, and FSDP — Uber explicitly chose DeepSpeed's ZeRO family over PyTorch's native FSDP, which is a position most of the academic benchmarks have been less decisive about. The writeup is light on code, but the architectural choices are specific enough to copy as a starting reference.

### 🎯 Takeaway (Hợp)
If a data platform is currently trying to decide between FSDP and DeepSpeed for a production LLM training workload, Uber's preference for ZeRO stage 3 with CPU offload is at least one data point grounded in a real Mixtral-scale deployment. The concrete next action is to run a small FSDP-vs-ZeRO ablation on your own model and storage layer before committing, because the 10 GB/s vs 3 GB/s network gap they report suggests the storage path matters more than the sharding framework on most clusters.

## 5. Espressif Ships First-Party C++ Components for Wi-Fi and Deep Sleep

| Field | Value |

### 🔭 Overview (Tổng)
Espressif's developer portal published "C++ Components for ESP-IDF" on 13 April 2026, introducing two reusable components — `wifi_manager` and `deep_sleep` — that encapsulate two of the most-repeated patterns in ESP32 firmware: provisioning Wi-Fi credentials through a captive portal and managing low-power sleep cycles with wake-up sources. The components are published as ESP-IDF add-ons through the standard component manager, which is the first-party packaging story that ESP32 developers have been asking for as an alternative to copying reference code out of `examples/`.

### 🔬 Details (Phân)
The `wifi_manager` component boots an access point with a login page, lets the user pick an SSID and enter credentials once, stores those credentials in non-volatile storage, and on subsequent boots reconnects automatically. It exposes a small C++ API for checking connection state and retrieving network info, which replaces the usual pile of event handlers around `esp_wifi_*`. The `deep_sleep` component wraps Espressif's sleep infrastructure with C++ primitives for timer and GPIO wake-up and a built-in boot counter stored in RTC memory, so power-budget calculations no longer require tracking sleep state manually.

Packaging the code as idf-components-registry entries rather than copy-paste snippets matters for fleet maintenance. It means a project `idf_component.yml` can pin to a version, and a future `idf.py update-dependencies` run will pick up security or bug fixes the same way ESP-IDF core does. It also gives vendors a more honest story about reproducible builds than "I took a sample and modified it a year ago". The release is narrowly scoped — two components, Wi-Fi and sleep — but it points toward a model where Espressif maintains common plumbing instead of leaving it to each project.

### 🎯 Takeaway (Hợp)
For a new ESP-IDF project starting this week, pull these two components in rather than porting captive-portal and sleep handling code out of a stale sample. For existing production firmware the switch is not urgent, but it is worth planning because the first-party path makes the next security bulletin easier to absorb. Đây cũng là tín hiệu để team DIY framework cân nhắc chuyển về component registry chính thức.

### 💡 Example
```cpp
// Minimal sketch combining wifi_manager and deep_sleep on ESP-IDF.
#include "wifi_manager.hpp"
#include "deep_sleep.hpp"

RTC_DATA_ATTR int bootCount = 0;

extern "C" void app_main(void) {
  Wifi wifi(std::string("wifi_manager"),
            std::string("ESP32-Portal"),
            std::string("en-US"));
  wifi.IsConnected();

  DeepSleep deepSleep(std::string("DeepSleep"), &bootCount);
  deepSleep.EnableTimerWakeup(30, "sec");
  deepSleep.GoToDeepSleep();
}
```
