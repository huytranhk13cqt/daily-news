## 1. Databricks Data Classification Goes GA With Agentic AI Tagging in Unity Catalog

### 🔭 Overview (Tổng)
Databricks promoted its Data Classification feature to general availability on 20 April 2026, after running it in beta since late 2025. The feature uses an agentic AI system built on a large language model to scan Unity Catalog tables, detect sensitive columns, and tag them so that governance policies such as attribute-based access control (ABAC) masks and filters apply automatically. For any team operating a lakehouse under GDPR, HIPAA, or a similar regime, this is a meaningful shift in how PII discovery is done — it moves from tag-by-hand or regex-driven scanners into the catalog itself.

### 🔬 Details (Phân)
Under the hood the classifier combines pattern recognition, metadata heuristics, and an LLM-based step to infer what a column represents, with Databricks reporting up to 60% higher accuracy than regex-only tools and up to 75% lower scan cost compared with the beta thanks to incremental scanning that only re-examines new or changed tables. Classification results land as tags on columns, which ABAC policies can then read to mask, tokenise, or outright block reads on rows that carry, say, `pii.email` or `pii.national_id`. The data stays within the customer's environment and follows Databricks' AI security standards, and storage of the results uses the default Unity Catalog storage with no additional billing.

Positioned against Snowflake's Horizon classifier, BigQuery's data profiler, and third-party tools like Immuta or Privacera, the move consolidates discovery and enforcement on the same governance surface. The practical win is that a platform team no longer has to stitch a separate classifier to Unity Catalog and keep two tag systems in sync — the classifier's output is the same tag dictionary that ABAC policies already consume. Caveats: LLM-based classifiers still misfire on ambiguous columns (initials, free-text notes, numeric IDs), so bulk approval UIs and spot audits remain part of the workflow, and organisations with strict residency rules should confirm where the model inference runs before enabling it in regulated workspaces.

### 🎯 Takeaway (Hợp)
Turn the feature on in a non-production catalog first, review the high-confidence detections in the Unity Catalog Data Classification UI, and only then wire the generated tags to ABAC masking policies. Với các team đã quen chạy regex scanner bên ngoài, đây là cơ hội để giảm số hệ thống cần maintain và giữ phân loại luôn mới sau mỗi schema change.

### 💡 Example
```sql
-- Once Data Classification tags columns, ABAC reads those tags.
-- Mask e-mail columns for everyone except the compliance group.
CREATE OR REPLACE FUNCTION catalog.sec.mask_email(e STRING)
RETURNS STRING
RETURN CASE
  WHEN is_account_group_member('compliance') THEN e
  ELSE regexp_replace(e, '(^.).*(@.*$)', '$1***$2')
END;

ALTER TABLE catalog.crm.customers
  ALTER COLUMN email
  SET MASK catalog.sec.mask_email
  USING TAGS ('pii.email');
```

## 2. Databricks Opens a Supervisor API for Building Agents in One Call

### 🔭 Overview (Tổng)
On the same 20 April 2026 release cut, Databricks opened a beta of the Supervisor API, a single-call interface for building agents on Mosaic AI. A caller supplies the model, the available tools, and the system instructions, and the platform handles tool selection, tool execution and response synthesis. The positioning is similar to AWS Bedrock Agents or Anthropic's agent primitives, but scoped to the Databricks data plane where tools can be SQL queries, Unity Catalog functions, or MCP servers registered through the company's AI Gateway.

### 🔬 Details (Phân)
Today the usual route to a production agent on Databricks involves orchestrating LangGraph or similar frameworks inside a Model Serving endpoint, packaging tool definitions by hand and writing the loop that decides when to call what. The Supervisor API collapses that into a declarative request: name the Mosaic model, list the tools, pass the instructions, and the platform runs the planning loop server-side. That has two consequences worth weighing. First, it removes a significant amount of glue code — which lowers the bar for data teams who want an agent inside a notebook without standing up a framework. Second, it moves control-flow decisions inside Databricks, which is helpful for governance and audit logging through the AI Gateway but less helpful if a team has invested in custom planners or multi-agent graphs.

Context matters here. The release arrived alongside Claude Opus 4.7 in Mosaic AI Model Serving and an AI Gateway for MCP servers in beta — all on the same 16–20 April 2026 wave. Databricks is clearly trying to make the Unity Catalog metadata the substrate for agent tools, so Supervisor API together with AI Gateway for MCP and the new Compute Log Delivery to Unity Catalog volumes form a single governance story rather than three unrelated features. It is still beta, so API surface, quota, and pricing remain subject to change before GA.

### 🎯 Takeaway (Hợp)
If a team is prototyping a data-agent use case and already has tools defined as Unity Catalog functions, the Supervisor API is worth an afternoon to compare against the in-house LangGraph pattern. The migration cost is small because the tool registry is shared, and the gain is audit trails that live next to the rest of the catalog. Cân nhắc giữ LangGraph nếu multi-agent planning là bắt buộc; ngược lại Supervisor API sẽ bớt boilerplate đáng kể.

### 💡 Example
```python
from databricks.agents import Supervisor

agent = Supervisor.create(
    model="databricks-claude-opus-4-7",
    tools=[
        "main.sales.refund_lookup",        # UC function
        "main.sales.issue_refund",         # UC function
        {"mcp_server": "tickets-prod"},    # via AI Gateway for MCP
    ],
    instructions=(
        "You are a refund triage assistant. Verify the order, "
        "check policy, and only call issue_refund after confirming."
    ),
)

reply = agent.invoke("Refund order #A-10294, customer says item arrived damaged.")
print(reply.output_text)
```

### 🔭 Overview (Tổng)
LILYGO unveiled the T-Watch Ultra on 20 April 2026, an IP65-rated smartwatch development board built around the ESP32-S3R8 with a 2.01-inch AMOLED, a Semtech SX1262 LoRa transceiver and a u-blox MIA-M10Q GNSS module. It is not a consumer watch aimed at fitness tracking; it is a wearable reference platform for LoRaWAN, meshtastic-style off-grid messaging, and GNSS trackers. At roughly $78 from the LILYGO store and $95 on AliExpress, it is priced as a hackable module rather than a finished product.

### 🔬 Details (Phân)
The board pairs a dual-core Tensilica LX7 at 240 MHz with 8 MB of PSRAM and 16 MB of QSPI flash, plus a microSD slot for local logging. The 410×502 AMOLED uses the CO5300 OSPI controller with 16.7 M colours and capacitive touch, and the package adds Wi-Fi 4, Bluetooth 5.0 LE, an ST25R3916 NFC front-end, a Bosch BHI260AP smart sensor for motion AI, a microphone, a haptic driver, an RTC chip, and a 1,100 mAh battery for around 4 Wh of capacity. The SX1262 supports 868, 915, and 920 MHz variants, which covers EU, US, and AU/NZ LoRa bands respectively.

What makes this interesting for IoT teams is the combination of LoRa, GNSS and a display in a single small form factor with a clear software story: Arduino IDE, PlatformIO, ESP-IDF and MicroPython all work out of the box. That covers both quick prototyping (Arduino, MicroPython) and production firmware paths (ESP-IDF), and the microphone plus BHI260AP allow sensor-fusion experiments that do not need a companion phone. Caveats: IP65 is splash-resistant rather than dive-proof, and the AMOLED plus SX1262 at 22 dBm will meaningfully reduce runtime compared with the battery's nominal capacity.

### 🎯 Takeaway (Hợp)
For anyone evaluating ESP32-S3 as a platform for wearable asset trackers, delivery-team tools, or field-service radios, the T-Watch Ultra is an honest starting point — production-adjacent hardware at a dev-board price. Plan the power budget around LoRa TX duty cycle rather than just processor load, because the display and radio will dominate the battery draw.

#include <RadioLib.h>
#include <SparkFun_u-blox_GNSS_v3.h>

SFE_UBLOX_GNSS gnss;          // u-blox MIA-M10Q over I2C
SX1262 radio = new Module(SS, DIO1, RST, BUSY);

void setup() {
  Wire.begin();
  gnss.begin();
  radio.begin(915.0 /* MHz, AU915 */, 125.0, 9, 7, 0x34, 22 /* dBm */);

void loop() {

             gnss.getLatitude() / 1e7, gnss.getLongitude() / 1e7);
    radio.transmit(msg);
  }
  esp_sleep_enable_timer_wakeup(60ULL * 1000000);
  esp_deep_sleep_start();
}

### 🔭 Overview (Tổng)
The BeagleBoard.org Foundation introduced BeagleConnect Zepto on 17 April 2026 and CNX Software amplified the coverage on 19 April. The headline number is a sub-one-dollar unit price targeting classroom and hobby use, but the more interesting detail for IoT engineers is the choice of Texas Instruments MSPM0L1117 Cortex-M0+ combined with mikroBUS-compatible friction-fit headers, two Qwiic connectors and a Zephyr-based SDK with MCUboot. The board is not a mass-market product yet — first production is slated for late summer 2026 — but it is aimed at being cheap enough that losing one in a field deployment is not a budget event.

### 🔬 Details (Phân)
The MSPM0L1117 runs an Arm Cortex-M0 at 32 MHz with 128 KB of flash and 16 KB of SRAM. That is modest but sufficient for a Zephyr application that reads a sensor, applies a small classifier or threshold, and forwards data over I2C or UART to a host. The mikroBUS headers open a library of roughly 2,000 off-the-shelf click boards, which matters because this kind of ecosystem access is usually the bottleneck for educational and rapid-prototype work rather than MCU performance. The two Qwiic connectors allow daisy-chaining with BeaglePlay, BeagleY-AI, BeagleBadge or any other Qwiic host, and a Raspberry Pi-style HAT connector exposes the board for SWD programming from a host single-board computer.

Firmware is where the real engineering story lives. BeagleBoard is shipping the board with a Zephyr RTOS SDK, a hard-to-brick MCUboot-based USB bootloader, and a BeagleConnect Greybus firmware that exposes the MCU's peripherals over I2C to a Linux or Zephyr host as if they were on-board. Greybus means that a Linux program on a BeaglePlay can talk to sensors attached to the Zepto without the developer writing any MCU firmware at all — the host kernel sees the remote peripheral as a local one. That is a non-trivial architectural choice because it lets a team separate "firmware work" from "application work" cleanly, which is rare at this price point. Target audience is educators, makers, and IoT researchers who want a disposable but standards-aligned endpoint.

### 🎯 Takeaway (Hợp)
Bookmark Zepto for curricula and for any project where the MCU price is a barrier — teaching labs, agricultural sensor fleets, one-off research deployments. The Greybus-over-I2C model is the part worth prototyping first, because it changes what "writing firmware" means and makes the board useful to teams that have Linux skills but no embedded C developer on staff.

### 💡 Example
```c
/* Minimal Zephyr app on Zepto: blink the RGB LED via Greybus-safe GPIO. */
#include <zephyr/kernel.h>
#include <zephyr/drivers/gpio.h>

#define LED_NODE DT_ALIAS(led0)
static const struct gpio_dt_spec led = GPIO_DT_SPEC_GET(LED_NODE, gpios);

int main(void) {
    gpio_pin_configure_dt(&led, GPIO_OUTPUT_ACTIVE);
    while (1) {
        gpio_pin_toggle_dt(&led);
        k_msleep(500);
    }
    return 0;
}
```
