## 1. Apache Airflow 2 Reaches End-of-Life as 3.2.1 Ships the Same Day

### 🔭 Overview (Tổng)
Apache Airflow 2 reached upstream end-of-life on 22/04/2026, the date previously pinned by the release plan back in 2024. From that cut-over, the 2.x line — including the final 2.11.x patch series — no longer receives security fixes, bug fixes, or provider updates from the Apache Software Foundation. The project shipped Apache Airflow 3.2.1 on the same calendar day, making the successor line the only supported target for upstream patches. For teams that still run 2.x in production, this is the moment when running unsupported software becomes an explicit risk register item rather than a future concern.

### 🔬 Details (Phân)
Airflow 3.2.1 is a maintenance release on top of 3.2.0 (07/04/2026) with roughly forty-five fixes covering scheduler, API, UI and core. The notable behavioural change is a permission tightening on the REST API: callers with read-only DAG access can no longer fetch full data from the `/dags` endpoint, because that response now also returns DAG run, task instance and human-in-the-loop detail objects. Accessing the endpoint now requires the additional `DAG_RUN`, `TASK_INSTANCE` and `HITL_DETAIL` read permissions. The UI theme configuration grew an optional `tokens` field that supports CSS-only overrides or an empty object to restore defaults. Internal bug fixes include logging configuration corrections, async engine disposal, session management, and N+1 query elimination in DAG serialization and task instance endpoints via bulk prefetch. OpenSearch log delivery is now a first-class integration.

Migration context matters here. Direct upgrades from Airflow 2.7+ to 3.x are supported, but the project recommends walking through 2.11 first to surface deprecation warnings before breaking changes bite. The 3.x line has already dropped SubDAGs, the SequentialExecutor default, and the legacy scheduler, and it mandates the new task-SDK-based authoring path. Astronomer is offering commercial support for Airflow 2 through April 2027, giving teams that cannot finish a migration in time a paid bridge — but the open-source project itself is done with 2.x.

### 🎯 Takeaway (Hợp)
Any team still shipping on Airflow 2.x should treat today as the start of a countdown: tag a migration project this week, pin Docker images at `apache/airflow:3.2.1`, and retire 2.x instances as soon as DAG compatibility is verified. Trì hoãn nghĩa là chấp nhận chạy một workload orchestrator mà không còn ai vá — với compliance (SOC 2, HIPAA, PCI-DSS) thì rủi ro này không nhỏ.

## 2. Snowflake Cortex Agents Generate SQL Directly From Semantic Views

| Field | Value |
|---|---|
| 📅 Date | 21/04/2026 |
| 🏷️ Topic | Release |
| 🌐 Domain | Data Engineering |
| 🔑 Keywords | `snowflake-cortex` · `cortex-agents` · `semantic-views` |
| ⚡ Impact | Medium |

### 🔭 Overview (Tổng)
Snowflake quietly flipped a significant behaviour in Cortex Agents on 21/04/2026: when a Cortex Analyst semantic view is attached as a tool, the agent now generates SQL directly instead of delegating that step to the Cortex Analyst service. The release notes frame it as an accuracy and latency improvement for analytical queries, which is the understatement version of what is actually an architectural change — one whole hop of the request path is gone. For any team building an NL-to-SQL assistant on Snowflake, this is the first change in months that touches the plumbing rather than the prompt.

### 🔬 Details (Phân)
Before the change, the agent path for a question like "what were gross sales by region last month" was a two-service handoff. The agent decided that the semantic view tool was relevant, called Cortex Analyst with the natural-language question, and Analyst ran its own NL-to-SQL pipeline over the view's metric and dimension metadata before returning rows. The agent then stitched those rows back into its answer. Each hop carried its own latency, its own retry semantics, and its own failure modes — for example, Analyst rate-limits separately from the agent endpoint. The new path flattens this: the agent reads the semantic view's definition — metrics, dimensions, joins — as grounded context and emits SQL itself against the Snowflake warehouse in the same turn.

Competitively, this lines Snowflake up with Databricks Genie's approach to AI/BI and with the dbt Semantic Layer MCP pattern that has become the default way to let an LLM talk to a governed metrics layer. The tradeoff worth flagging is that the agent is now directly responsible for SQL correctness; Cortex Analyst, which had its own tuning loop over the semantic layer, is no longer the gatekeeper. Regressions in SQL correctness land in the agent's evaluation harness rather than in Analyst's, so teams should re-run their offline benchmarks before declaring parity.

### 🎯 Takeaway (Hợp)
If an analytics-assistant project is already wired to Cortex Agents with Analyst tools, re-benchmark latency and answer accuracy on the new direct-SQL path this sprint — the win is usually real but not universal. For new projects, start directly on the new path; the old two-hop configuration is a legacy shape at this point.

### 💡 Example
```sql
-- With 21/04/2026 release, a Cortex Agent that owns a semantic view
-- emits SQL itself. Register the agent and attach the semantic view
-- as a tool — no separate Cortex Analyst call is needed at runtime.
CREATE OR REPLACE CORTEX AGENT sales_copilot
  WITH MODEL = 'claude-opus-4-7'
  TOOLS = (
    SEMANTIC_VIEW => analytics.public.sales_semantic_view,
    SEARCH_SERVICE => analytics.public.sales_docs_search
  )
  INSTRUCTIONS = $$
    You answer questions about monthly and quarterly sales. Always
    restrict queries to the fiscal_period the user specifies. If a
    metric is not defined in the semantic view, refuse rather than
    improvising a calculation.
  $$;

-- Invoke it. The agent plans and generates SQL itself against the
-- semantic view, then returns both the SQL and the result rows.
SELECT CORTEX.AGENT_COMPLETE(
  'sales_copilot',
  'Gross sales by region for Q1 FY26, ordered high to low.'
);
```

## 3. AWS IoT Greengrass v2.17 Adds Non-Root Runtime and HSM-Backed Nucleus Lite

| Field | Value |
|---|---|

| 🏷️ Topic | Release |
| 🌐 Domain | IoT |
| 🔑 Keywords | `aws-iot-greengrass` · `nucleus-lite` · `pkcs11` |
| ⚡ Impact | Medium |

### 🔭 Overview (Tổng)
AWS released IoT Greengrass v2.17 on 20/04/2026. The release is unusually dense for a Greengrass minor: the edge runtime can now install and run as a non-root user on Linux, a new "nucleus lite" variant cuts memory consumption dramatically, and cryptographic material can live in a hardware security module via a PKCS#11 interface. Fleet Provisioning picks up TPM 2.0 support, and components gain an uninstall lifecycle hook that fires when a component is removed from a device. None of the individual pieces are novel in the wider IoT platform space, but in aggregate they close a multi-year gap against Azure IoT Edge and Balena.

### 🔬 Details (Phân)
The non-root install is the change with the broadest reach. Enterprise deployments in regulated environments — healthcare, utilities, financial branch offices — routinely forbid root-owned long-running processes by policy, which until now forced teams to use Azure IoT Edge or to fight with `sudo`-owning wrappers around Greengrass. v2.17 supports running the nucleus, components and deployment logic under an unprivileged service account, with a narrowly scoped capability set for network operations. Memory is the other highlight: the Secure Tunneling lite component drops to roughly 4MB of memory, compared to 36MB for the existing standard component, which finally makes Greengrass plausible on sub-gigabyte edge gateways. The uninstall lifecycle hook runs a user-supplied script when the device receives a deployment that removes a component, which finally gives operators a clean way to clean up configuration files, kernel modules or systemd units without writing a separate cron job.

Hardware identity also gets a real answer. PKCS#11 lets the nucleus lite authenticate to AWS IoT Core using private keys stored in an HSM — an ATECC608, a Microchip SE050, or a TPM 2.0 — rather than in a filesystem. Fleet Provisioning's new TPM 2.0 flow lets the device negotiate its initial certificate without the private key ever leaving silicon. The practical upshot is that an attacker who lifts firmware from a device no longer automatically walks away with a usable IoT identity.

### 🎯 Takeaway (Hợp)
Existing Greengrass fleets running as root should budget a pilot migration to non-root plus PKCS#11-backed identity this quarter; it is a noticeable compliance uplift for relatively small effort. New gateways should default to v2.17 with nucleus lite components from day one and avoid carrying forward the 36MB standard-component footprint.

### 💡 Example
```bash
# Install Greengrass v2.17 as a non-root service account on Linux.
sudo useradd --system --no-create-home --shell /usr/sbin/nologin ggc_user
sudo groupadd --system ggc_group
sudo java -jar ./GreengrassInstaller.jar \
  --aws-region ap-southeast-2 \
  --thing-name edge-gw-042 \
  --thing-group-name factory-floor \
  --component-default-user ggc_user:ggc_group \
  --setup-system-service true \
  --run-with-non-root-user true

# Switch the nucleus lite to HSM-backed identity via PKCS#11.
cat <<'YAML' > /greengrass/v2/config/pkcs11.yaml
aws.greengrass.crypto.Pkcs11Provider:
  configuration:
    name: "softhsm"
    library: "/usr/lib/softhsm/libsofthsm2.so"
    slot: 0
    userPin: "1234"
system:
  privateKeyPath: "pkcs11:object=iot-device-key;type=private"
  certificateFilePath: "pkcs11:object=iot-device-cert;type=cert"
YAML
sudo systemctl restart greengrass.service
```

### 🔭 Overview (Tổng)
CISA added eight vulnerabilities to its Known Exploited Vulnerabilities catalog on 20/04/2026, citing evidence of active exploitation. Three of them are Cisco Catalyst SD-WAN Manager bugs — CVE-2026-20122, CVE-2026-20128 and CVE-2026-20133 — which together cover arbitrary file write, recoverable password storage, and unauthorised information exposure. Federal Civilian Executive Branch agencies have an accelerated remediation deadline of 23/04/2026, which is today in Perth. The remaining five (PaperCut NG/MF, JetBrains TeamCity, Kentico Xperience, Quest KACE SMA, Synacor Zimbra) carry the standard 04/05/2026 deadline. SD-WAN Manager sits at the branch-office edge and orchestrates WAN overlay configuration across downstream IoT and OT sites, which is why this lands on an IoT digest rather than a generic IT one.

### 🔬 Details (Phân)
CVE-2026-20122 (CVSS 5.4) is an incorrect privileged-API usage flaw that lets an authenticated actor upload and overwrite arbitrary files on the Manager host. CVE-2026-20128 (CVSS 7.5) is a recoverable password storage weakness — credentials are stored in a form that allows retrieval, which an attacker with any foothold on the Manager can convert into Data Collection Agent (DCA) user privileges and from there into downstream overlay configuration. CVE-2026-20133 (CVSS 6.5) exposes sensitive information to an unauthorised actor, most plausibly through an under-authorised REST endpoint. None of the three have a CVSS above 8, but the KEV addition is the key signal: CISA only adds vulnerabilities with evidence of exploitation in the wild. Treat the score as the floor, not the ceiling.

The broader context is that SD-WAN Manager is the control plane for branch and industrial-edge connectivity at the scale of thousands of devices per customer. A Manager compromise lets an attacker push policy down to remote sites, reroute traffic through adversary infrastructure, or pre-position malicious configuration on edge routers that terminate into IoT and OT VLANs. This is the same class of risk as the 2024 and 2025 Cisco IOS-XE management-plane incidents, and the KEV inclusion suggests the same operator playbooks are being re-used. The other five CVEs on the list, including the CVSS 10.0 Quest KACE SMA authentication bypass, are not edge-specific but are worth patching on the standard deadline.

### 🎯 Takeaway (Hợp)
Operators of Cisco Catalyst SD-WAN Manager should apply the current fixed releases today, rotate any DCA-tier credentials stored on the Manager, and audit overlay configuration for unauthorised pushes since the bugs' disclosure window. For non-federal shops the federal deadline is not binding, but the active-exploitation label makes this a patch-now item regardless of who is asking.
