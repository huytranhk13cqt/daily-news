## 1. Databricks Lakeflow Designer Enters Public Preview With Drag-and-Drop and Natural-Language Pipeline Authoring

### 🔭 Overview (Tổng)
Databricks flipped Lakeflow Designer into Public Preview on 22/04/2026 as part of its April platform release. The feature is a visual transformation authoring surface that sits on top of the Lakeflow pipeline runtime introduced last year. Users can assemble a pipeline via a drag-and-drop canvas of sources, transforms and sinks, and can ask a built-in natural-language assistant to emit the underlying Lakeflow code. It targets the same gap that dbt Cloud's graphical editors and Snowflake Openflow's visual builder address — analysts who want to build transformations without writing SQL by hand, but who still need the result to land as versioned, testable code.

### 🔬 Details (Phân)
The Designer is available in Public Preview on AWS, Azure and GCP-hosted Databricks workspaces. Pipelines authored in the Designer compile to the same Lakeflow definitions that the CLI and the SDK produce, which means a pipeline built visually can be exported to a Git repository, version-controlled, reviewed as code, and deployed with Databricks Asset Bundles. The natural-language entry point accepts prompts like "filter orders to the last 30 days and join to customers on customer_id" and converts them into canvas nodes whose generated SQL is editable in a side panel. The same release adds a language selector for AI-generated comments in Catalog Explorer, which persists per browser and applies across entity types — a small but related polish on the agentic authoring surface.

Positioning matters here. Databricks' previous visual authoring option was the Lakeview-era notebook plus the older DLT pipeline UI, both of which produced artefacts that were awkward to reconcile with code-first workflows. Lakeflow Designer is the first Databricks-authored visual tool where the canvas is a thin presentation layer over a canonical Lakeflow definition, not a separate mode. The tradeoff worth flagging is that Public Preview in Databricks language means no SLA and no backward-compatibility guarantee, so teams should keep generated pipelines in a "preview" namespace and avoid making them critical-path until GA. The AI-generated-comment language selector is a Catalog Explorer feature and does not yet extend to dashboards or Genie spaces.

### 🎯 Takeaway (Hợp)
Analytics teams that currently jump between a pure-code dbt project and a notebook should pilot Lakeflow Designer on a non-critical workload this sprint to see whether the "visually authored, code-reviewed" round-trip actually works for their approvers. For code-first platform teams, the relevant question is governance, not productivity — Git review, CI and environment promotion should all stay exactly as they were, which is the point of a Designer that emits canonical Lakeflow code.

```yaml
# A Lakeflow Designer pipeline round-trips to a Lakeflow definition
# that can be committed to Git and deployed with Databricks Asset Bundles.
# orders_curated.yml — generated from the 22/04/2026 Designer preview.
pipelines:
  orders_curated:
    catalog: main
    target: sales_curated
    libraries:
      - file:
          path: ./transformations/orders_recent.sql
    configuration:
      bundle.mode: development
      pipelines.trigger.interval: "10 minutes"

# transformations/orders_recent.sql
CREATE OR REFRESH STREAMING TABLE orders_recent AS
SELECT o.order_id, o.customer_id, o.order_ts, c.country
FROM   STREAM(main.raw.orders) o
JOIN   main.dim.customers c USING (customer_id)
WHERE  o.order_ts >= current_timestamp() - INTERVAL 30 DAYS;
```

## 2. Spring for Apache Kafka 4.1.0-RC1 Adds Share-Consumer Ack Modes and Streams Rebalance Protocol Selection

| Field | Value |
|---|---|
| 📅 Date | 22/04/2026 |
| 🏷️ Topic | Release |
| 🌐 Domain | Data Engineering |
| 🔑 Keywords | `spring-kafka` · `share-consumer` · `kafka-streams` |
| ⚡ Impact | Medium |

### 🔭 Overview (Tổng)
The Spring for Apache Kafka team shipped three versions on 22/04/2026: the 4.1.0-RC1 release candidate alongside maintenance releases 4.0.5 and 3.3.15. The 4.1 line is the library's first meaningful follow-up to the Apache Kafka 4.x baseline and leans heavily on the KIP-932 share-consumer API and the new Streams rebalance protocol that landed upstream. For teams running Kafka clients through Spring Boot — which in practice is most of the Java-on-Kafka population — this is the release that starts exposing "queues over Kafka" to idiomatic Spring code.

### 🔬 Details (Phân)
The headline change in 4.1.0-RC1 is how share consumers acknowledge. The previous boolean `autoCommit` style has been retired in favour of a `ShareAckMode` enum with three values: `EXPLICIT`, `MANUAL` and `IMPLICIT`. Explicit acks are surfaced through a new `AcknowledgementCommitCallback` that returns asynchronously, so a consumer can continue processing while a prior commit is in flight. The listener container grew new lifecycle events for stop and failure scenarios so that outer frameworks — in particular Spring Cloud Stream — can react to share-consumer transitions without polling internal state.

On the Kafka Streams side, Spring now exposes the `groupProtocol` configuration so that applications can opt into the classic JoinGroup-era protocol or the server-side rebalance protocol shipped in Apache Kafka 4.1. The release also picks up KIP-1034-aligned native dead-letter-queue handling for Streams and ships two new exception handlers — `RecoveringProcessingExceptionHandler` and `RecoveringProductionExceptionHandler` — that let an application continue past a failing record instead of killing the thread. The version matrix pins 4.1.0-RC1 to Spring Boot 4.1.0-RC1, 4.0.5 to Spring Boot 4.0.6, and 3.3.15 to Spring Boot 3.5.14; the 4.0.5 and 3.3.15 patches carry filtering, startup and immutability fixes, plus framework bumps.

### 🎯 Takeaway (Hợp)
Teams that already build on Spring Boot 4.0 should upgrade to 4.0.5 during their next release window — it is a patch and the risk profile is low. Teams eyeing KIP-932 share consumers for queue-style workloads should stand up a 4.1.0-RC1 sandbox now so the ack-mode migration surface is understood well before GA, particularly the `ShareAckMode` enum change which is not source-compatible with the deprecated boolean.

### 💡 Example
```java
// Spring Kafka 4.1.0-RC1 share consumer with the new ShareAckMode API.
@KafkaListener(
    topics = "orders",
    groupId = "orders-workers",
    containerFactory = "shareContainerFactory"
)
public void handle(ConsumerRecord<String, Order> rec, Acknowledgment ack) {
    try {
        process(rec.value());
        ack.acknowledge();           // explicit success
    } catch (RetryableException ex) {
        ack.nack(Duration.ofSeconds(5));
    }
}

@Bean
ConcurrentMessageListenerContainerFactory<String, Order> shareContainerFactory(
        ConsumerFactory<String, Order> cf) {
    var f = new ConcurrentMessageListenerContainerFactory<String, Order>();
    f.setConsumerFactory(cf);
    f.getContainerProperties().setShareAckMode(ShareAckMode.EXPLICIT);
    f.getContainerProperties().setCommitCallback(
        (offsets, ex) -> {
            if (ex != null) log.warn("share commit failed", ex);
        });
    return f;
}
```

## 3. Databricks Data Classification Reaches GA With Agentic Tagging Across Unity Catalog

| Field | Value |
|---|---|
| 📅 Date | 20/04/2026 |
| 🏷️ Topic | Release |
| 🌐 Domain | Data Engineering |
| 🔑 Keywords | `unity-catalog` · `data-classification` · `pii-tagging` |
| ⚡ Impact | Medium |

### 🔭 Overview (Tổng)
Databricks Data Classification moved from Public Preview to General Availability on 20/04/2026. The feature scans tables registered in Unity Catalog and attaches column-level tags for sensitive data classes — personally identifiable information, payment data, regulated health data — using an agentic AI system rather than a rules-only classifier. GA status means the feature now carries the standard Databricks product SLA and is intended to be wired into production governance workflows rather than just explored. It lands into a market where Snowflake's sensitivity classifications, Google Cloud DLP, and AWS Glue Sensitive Data Detection have been the previous defaults.

### 🔬 Details (Phân)
The classifier runs as a managed scheduled job inside a workspace and walks the catalog's tables on a configurable cadence. Rather than matching columns purely on regex patterns, the agentic layer examines table metadata — column name, description, position, the surrounding schema — alongside sampled values and then calls an internal model to decide whether a column is, for example, `email`, `government_id`, `payment_card_number` or not sensitive at all. Detected classes are written back as Unity Catalog tags, which means they immediately become queryable through `information_schema.tag_references` and can drive row filters, column masks, and Attribute-Based Access Control rules.

Operationally, tags set by Data Classification are namespaced and immutable from user UPDATEs, which prevents a downstream user from silently unflagging a column. Administrators can still override a detection via an explicit acknowledgement, and the agent's confidence score is exposed on the tag for audit. The GA release also adds region-specific rollouts and documents the scanning cost model — a non-trivial consideration for catalogs with thousands of tables. In parallel on 20/04, Databricks shipped the Supervisor API for Building Agents in Beta, which targets a different slice of agent construction and is worth a separate evaluation.

### 🎯 Takeaway (Hợp)
Governance and security teams that have been running Data Classification in Preview should schedule the promotion to GA and update their quarterly audit procedures to consume the new immutable tags directly, rather than maintaining a parallel regex-based inventory. Teams still on a home-grown scanner should trial the GA version on a single catalog before committing — the quality gain is real, but so is the compute bill if scan frequency is set aggressively across a large metastore.

### 💡 Example
```sql
-- Inspect what Data Classification has tagged after promotion to GA.
SELECT t.catalog_name, t.schema_name, t.table_name, t.column_name,
       t.tag_name, t.tag_value
FROM   system.information_schema.column_tags t
WHERE  t.tag_name LIKE 'system.data_classification.%'
  AND  t.catalog_name = 'main'
ORDER  BY 1, 2, 3, 4;

-- Drive an ABAC row filter off the immutable classification tag.
CREATE OR REPLACE FUNCTION main.sec.hide_pii(region STRING) RETURN BOOLEAN
RETURN current_user() IN (SELECT principal FROM main.sec.pii_readers)
    OR region = 'NONPII';

ALTER TABLE main.sales.orders
  SET ROW FILTER main.sec.hide_pii ON (region);
```

## 4. Akamai Catches First In-the-Wild Mirai Campaign Against EoL D-Link DIR-823X via CVE-2025-29635

| Field | Value |
|---|---|
| 📅 Date | 22/04/2026 |
| 🏷️ Topic | Security |
| 🌐 Domain | IoT |
| 🔑 Keywords | `mirai-botnet` · `d-link-dir-823x` · `cve-2025-29635` |
| ⚡ Impact | High |

### 🔭 Overview (Tổng)
Akamai's Security Intelligence and Response Team published evidence on 21/04/2026 — amplified across the press on 22/04 — of the first in-the-wild exploitation of CVE-2025-29635, a command-injection flaw in D-Link DIR-823X series routers that had been publicly disclosed thirteen months earlier. The payload is a fresh Mirai variant that Akamai tracks as "tuxnokill". The affected routers were declared end-of-life by D-Link in September 2025, which means there is no vendor patch in flight and will not be one; mitigation is a combination of firewalling, network segmentation and replacement. In parallel, Fortinet reported a separate Mirai variant, "Nexcorium", hitting TBK DVRs via CVE-2024-3721, which together bracket a noticeable uptick in Mirai activity against long-abandoned IoT hardware.

### 🔬 Details (Phân)
CVE-2025-29635 is a command injection in the HTTP admin interface of DIR-823X firmware 240126 and 24082. The vulnerable path is a POST to `/goform/set_prohibiting`, where an attacker-controlled `macaddr` value flows unsanitised into a `snprintf` buffer that is later handed to a shell invocation; the result is arbitrary command execution without authentication against the router's management interface. The tuxnokill payload is a standard Mirai lineage binary compiled for ARMv7, MIPS and other embedded Linux architectures, XOR-encoded with key `0x30`, and supports the usual Mirai DDoS primitives — TCP SYN/ACK floods, UDP floods, HTTP NULL. The binary phones home to C2 at `64.89.161.130:44300` and fetches the initial stage from `88.214.20.14`. Akamai's honeypots first captured exploitation attempts in early March 2026 and the 21/04 writeup lists five SHA-256 hashes for the loader and variant binaries.

Two pieces of context matter operationally. First, the routers are classified EoL, so the only vendor-supplied action is "replace" — treat a DIR-823X that is still public-facing as a compromised host by default. Second, the public PoC has been available since March 2025, which is the lesson Akamai emphasises: a disclosure without exploitation is not a state the industry should rely on, and end-of-life IoT on residential ASNs remains a renewable botnet resource. Fortinet's Nexcorium variant behaves the same way against TBK DVRs and reuses CVE-2017-17215 against older Huawei routers for good measure, with four persistence vectors (`/etc/inittab`, `/etc/rc.local`, a systemd unit and a crontab entry) and a self-deleting dropper.

### 🎯 Takeaway (Hợp)
Operators and ISPs should block inbound TCP/44300 egress and the two reported IPs at the perimeter, and sweep customer premise equipment fleets for DIR-823X devices that still expose the admin UI to the WAN. For home-lab and SMB operators, any DIR-823X on a public IP is a compromise-or-imminent-compromise case; replace rather than patch, because no patch is coming.

### 💡 Example
```bash
# Blunt short-term mitigation for a DIR-823X still on the network: block
# the admin HTTP interface and the known tuxnokill C2 egress.
iptables -I INPUT  -p tcp --dport 80   -i wan0 -j DROP
iptables -I OUTPUT -p tcp -d 64.89.161.130    -j DROP
iptables -I OUTPUT -p tcp --dport 44300       -j DROP

# Quick hunt for tuxnokill persistence traces on a Linux embedded host.
grep -E 'tuxnokill|AI\.NEEDS\.TO\.DIE'   /etc/rc.local /etc/inittab 2>/dev/null
find /etc/systemd/system -name '*.service' -newer /var/log/messages -print
crontab -l | grep -E 'curl|wget|tftp'
```
