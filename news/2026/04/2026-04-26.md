### 🔭 Overview (Tổng)
Databricks moved Lakeflow Designer to public preview on 23/04/2026, slotting a visual, no-code pipeline builder into the existing Lakeflow product line that already covers ingestion, transforms and orchestration. The pitch is squarely at analysts who today live in spreadsheets and brittle BI extracts, not at the data engineers who already write Python or SQL on the platform. The relevance for a DE team is governance: every drag-and-drop step compiles to versioned Python that lands inside Unity Catalog with the same lineage, permissions and observability as a hand-written pipeline. That changes the political calculation around whether to let business users self-serve, because shadow ETL stops being the only alternative.

### 🔬 Details (Phân)
Lakeflow Designer's canvas is built on top of Genie Code, the agentic coding assistant Databricks shipped earlier in 2026, so the natural-language prompts that describe a transform are answered with code that reads Unity Catalog metadata for grounding. Each visual operator — join, filter, pivot, window, deduplicate — generates a real Python expression that the user can inspect, edit and check into Git. Step-by-step previews run on sampled data so an analyst can see exactly what a transform did before extending the pipeline. The output is a Lakeflow job that can be scheduled, parameterised and monitored alongside engineer-authored jobs, rather than a separate runtime. Compute is billed against existing Databricks SQL or general-purpose clusters, so there is no per-seat fee for Designer itself — a deliberate contrast with Tableau Prep, Alteryx and the seat-priced low-code competitors.

The competitive frame matters. Snowflake's Sigma partnership and the dbt Cloud Visual Editor target the same analyst persona, but both keep transforms outside the warehouse's primary governance plane until the analyst hits "promote." Lakeflow Designer's approach is to refuse that gap: a Designer pipeline is a Lakeflow pipeline, full stop. The trade-off is platform lock-in on the modelling side, since the generated code uses Lakeflow's declarative pipeline API rather than vanilla PySpark, and an analyst's pipeline graduates into something a DE team must adopt and maintain rather than rewrite.

### 🎯 Takeaway (Hợp)
DE teams on Databricks should pilot Designer with a small analyst group this sprint and codify two rules up front: which Unity Catalog schemas analysts can write into, and a review gate before a Designer pipeline is promoted to a production schedule. Without those guardrails the no-code surface tends to multiply faster than engineering can govern it.

### 💡 Example
```python
# A Lakeflow Designer pipeline compiles to declarative Lakeflow Python.
# The same code can be edited by hand and committed alongside
# engineer-authored pipelines.
import dlt
from pyspark.sql.functions import col, sum as _sum

@dlt.table(
    name="orders_clean",
    comment="Drop test rows; designer step generated this filter.",
)
def orders_clean():
    return (
        spark.read.table("retail.bronze.orders")
        .filter(col("customer_id").isNotNull())
        .filter(~col("order_id").startswith("TEST-"))
    )

@dlt.table(name="sales_by_region")
def sales_by_region():
    return (
        dlt.read("orders_clean")
        .groupBy("region")
        .agg(_sum("gross_amount").alias("gross_sales"))
    )
```

### 🔭 Overview (Tổng)
Snowflake's 23/04/2026 press release pulls Cortex Code and Snowflake Intelligence into a single positioning: a control plane for "the agentic enterprise" that reaches outside the warehouse. The product news under that umbrella matters more than the marketing wrapper. Cortex Code now speaks both Anthropic-style Model Context Protocol (MCP) and Agent Communication Protocol (ACP), and ships first-party connectors to AWS Glue, Databricks and PostgreSQL. Snowflake Intelligence — the business-user assistant that learns user preferences against governed data — picks up MCP connectors for Gmail, Google Calendar, Google Docs, Jira, Salesforce and Slack. The cumulative effect is that an agent built in or against Snowflake no longer has to land everything in a Snowflake table to be useful.

### 🔬 Details (Phân)
The MCP additions matter because Cortex Code can now act as either an MCP server (exposing Snowflake-native tools to a foreign agent runtime such as Claude Code or VS Code Copilot) or an MCP client (reaching out to external tools). The ACP plumbing extends that to agent-to-agent traffic, which is the piece the simpler MCP pattern does not cover. In practice this means a multi-system workflow — say, a credit decisioning agent that queries Snowflake-resident features, calls a Postgres customer-history tool, and asks a Databricks ML model for a probability — can be composed without Snowflake hosting every step. Cortex Code also gains IDE plugins for Visual Studio Code and a dedicated plugin for Claude Code, so the day-to-day surface for builders is the editor rather than the Snowflake console.

The honest read on the announcement is that Snowflake is conceding what was already true: enterprises run on multiple data platforms, and the agent layer needs to traverse them. That puts Snowflake in direct contact with Databricks's own Mosaic AI agent stack and with the protocol-level work Anthropic and Google are doing on MCP and A2A respectively. Where Snowflake wins is on governance — identity, policy, lineage and Horizon catalog — provided customers funnel agent traffic through Snowflake's audited paths rather than around them. Where it risks losing is performance: every external MCP hop is a network round-trip, and ACP-mediated agent coordination can be hard to debug when something stalls.

### 🎯 Takeaway (Hợp)
Teams already shipping Cortex Agents should treat this as the moment to formalise their MCP server registry and inventory which external tools actually need agent access — uncontrolled MCP sprawl is the new shadow IT. For new agentic projects the practical play is to start with one external system (Postgres or Glue), measure end-to-end latency and trace fidelity, and only then expand the surface.

## 3. CISA Adds Four CVEs to KEV Spanning Samsung MagicINFO SimpleHelp and a D-Link Router

### 🔭 Overview (Tổng)
CISA added four vulnerabilities to its Known Exploited Vulnerabilities catalog on 24/04/2026, four days after the eight-CVE batch that included the Cisco SD-WAN Manager bugs covered in the 23/04 digest. The new additions cover Samsung MagicINFO 9 Server (CVE-2024-7399), two SimpleHelp remote-support flaws (CVE-2024-57726 and CVE-2024-57728) and a D-Link DIR-823X command-injection bug (CVE-2025-29635). All four carry the standard 8 May 2026 federal remediation deadline under BOD 22-01. The IoT framing on this one is deliberate: MagicINFO drives digital signage and kiosk fleets, the DIR-823X is a consumer/SoHo router family that shows up in branch deployments, and SimpleHelp is the remote-support backdoor used to manage edge devices in the field.

### 🔬 Details (Phân)
CVE-2024-7399 is a path-traversal flaw in Samsung MagicINFO 9 Server before version 21.1050 that allows an unauthenticated attacker to write arbitrary files with system privileges — useful as a foothold to drop a webshell or pivot into the signage network. The bug is not new to the catalogue; it was first flagged as exploited in mid-2025, and the fact that CISA is adding it almost a year later signals continued in-the-wild abuse against unpatched fleets. The two SimpleHelp CVEs are a chain: CVE-2024-57726 is a missing-authorisation flaw that lets a low-privileged technician forge API keys with elevated scopes, and CVE-2024-57728 is a path traversal that an attacker with those keys can use to read or write arbitrary files. Together they get an attacker from a compromised technician account to full server admin. CVE-2025-29635 in the D-Link DIR-823X is a command-injection bug; the device family is end-of-life, and there is no fix, so the only defensible action is replacement or hard isolation. The same CVE has been linked to the "tuxnokill" Mirai variant currently rounding up DIR-823X units into a DDoS botnet.

The broader pattern is that two of the four — the MagicINFO bug and the D-Link bug — sit on devices that nobody patches: signage controllers and SoHo routers. The KEV listing is effectively CISA telling federal asset owners to find them and either patch, replace or air-gap them by 8 May. The remote-support angle on SimpleHelp is also worth flagging because the chain is the kind of finding that lands on regional MSP infrastructure and from there into hundreds of downstream customer fleets.

### 🎯 Takeaway (Hợp)
Audit any environment that runs Samsung MagicINFO, SimpleHelp Server, or end-of-life D-Link DIR-823X kit this week, and treat the 8 May federal date as a reasonable internal deadline regardless of whether BOD 22-01 binds. For DIR-823X specifically the only safe answer is removal — không có bản vá nào sẽ tới.

## 4. FIRESTARTER Backdoor on Cisco ASA Survives Patching and Only Yields to a Power Cycle

| Field | Value |
|---|---|
| 📅 Date | 23/04/2026 |
| 🏷️ Topic | Security |
| 🌐 Domain | IoT |
| 🔑 Keywords | `cisco-asa` · `firestarter-backdoor` · `cisa-ncsc` |
| ⚡ Impact | High |

### 🔭 Overview (Tổng)
On 23/04/2026 CISA published malware analysis report AR26-113A on FIRESTARTER, a Linux ELF backdoor found running on Cisco Adaptive Security Appliance (ASA) and Firepower Threat Defense (FTD) devices. The UK NCSC co-signed the report. The analysed sample was recovered from a federal civilian agency device that had been compromised through CVE-2025-20333 and CVE-2025-20362, the ArcaneDoor-class ASA flaws disclosed in late 2025 and patched in September 2025. The headline finding is that FIRESTARTER persists across the very patches that closed the initial-access bugs, so a fleet that "patched in September" is not necessarily clean today. The only documented removal path is a hard power-cycle of the device, because the persistence mechanism lives entirely in volatile memory but is re-installed on every warm restart from a tampered FXOS component.

### 🔬 Details (Phân)
The FIRESTARTER ELF runs on the FXOS base operating system that underpins ASA and FTD on Firepower hardware. CISA's analysis describes it as a thin C2 shim — its primary job is to keep a beacon alive to attacker infrastructure and to load follow-on stages on demand, rather than to carry rich functionality on disk. Persistence is the interesting part: the implant rewrites a boot-time loader so that any reboot that does not interrupt power simply re-stages the in-memory backdoor. A `reload` issued from the ASA CLI is therefore not enough. The remediation guidance is explicit — physically remove power, then re-image — which is operationally awkward for branch and edge deployments where physical access is rare. CISA published YARA rules that operators can run against a disk image or core dump to identify the backdoor without booting the device, and these are the recommended detection path before assuming a device is clean.

The broader context is that FIRESTARTER appears to be the operational sequel to the ArcaneDoor campaign first reported in 2024, which used the same class of edge-device targeting against perimeter firewalls. Cisco's Talos team and CISA both tie this activity to a state-aligned APT actor whose tradecraft prioritises long-dwell, low-noise access to network plumbing. The reason this lands on an IoT/embedded digest rather than a generic IT one is that ASA and FTD on Firepower hardware are firmware-managed appliances at the security perimeter — closer in lifecycle and update model to industrial gateways than to a general-purpose Linux host. Every lesson FIRESTARTER teaches about patch verification on appliances applies equally to OT routers, building-management gateways and field-installed industrial firewalls.

### 🎯 Takeaway (Hợp)
Operators of Cisco ASA or FTD on Firepower hardware should run CISA's YARA rules against current device images this week, and schedule a physical power-cycle plus re-image for any device that flagged. Treat the September 2025 patch alone as insufficient evidence of a clean state.

### 💡 Example
```bash
# Triage workflow for an ASA/FTD device suspected of FIRESTARTER persistence.

# 1. Pull a core dump WITHOUT rebooting the device first; rebooting can
#    reload the backdoor before evidence is captured.
ciscoasa# verify /sha-512 disk0:/coredumpfsys/core.gz
ciscoasa# copy disk0:/coredumpfsys/core.gz tftp://10.0.0.5/core-fw01.gz

# 2. Run CISA's YARA rule set against the core dump on a forensics host.
yara -r /opt/cisa-yara/firestarter.yar /forensics/core-fw01.gz

# 3. If FIRESTARTER hits, the documented remediation is power-off,
#    re-image with a known-good FXOS bundle, then re-apply config.
#    A `reload` is NOT sufficient.
ciscoasa# write standby
# (then physically remove power for >30 seconds before re-imaging)
```
